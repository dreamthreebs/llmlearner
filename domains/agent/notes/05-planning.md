## 核心问题

**ReAct 一步只看一步，复杂任务怎么办？**

ReAct 的工作方式是：想一步、做一步、看结果、再想下一步。对于简单任务（2-5 步可完成的），这足够了。但面对复杂任务——比如"帮我写一个有用户认证的 Web 应用"——没有全局规划，Agent 很容易迷失方向、重复劳动、或走入死胡同。

规划就是解决这个问题：**先想清楚要做什么，再一步步执行**。

> 可靠程度：Level 2-3（规划是 Agent 的核心能力之一，但什么是最好的规划策略仍是开放问题）

## 1. 为什么 ReAct 不够：无规划的失败模式

### 1.1 典型失败场景

```
任务："分析某公司过去 5 年的财报数据，找出增长趋势，并生成一份报告。"

ReAct 风格（无全局规划）：
  Thought 1: 我应该先搜索这家公司的财报。
  Action 1: search("公司 X 财报")
  Observation 1: 找到了 2024 年的财报...

  Thought 2: 我需要更多年份的数据。
  Action 2: search("公司 X 2023 财报")
  Observation 2: 找到了...

  Thought 3: 继续搜索 2022 年...
  ...
  [搜索了 5 轮之后]

  Thought 6: 我好像应该开始分析了... 但我忘了用户想要什么格式的报告。
  [回头看对话历史，发现上下文已经很长了]

  Thought 7: 我先分析一下增长率... 等等，我之前搜到的 2021 年数据在哪？
  [在长长的对话历史中找不到了]
```

无规划的 Agent 在长任务中的三个典型问题：
1. **目标遗忘**：执行了太多步之后忘了最终目标
2. **重复劳动**：做过的事又做一遍
3. **缺乏结构**：不知道先做什么后做什么，随机探索

### 1.2 人类怎么做

面对同一个任务，一个有经验的人会：

```
1. 确定需要的数据：2020-2024 年财报
2. 收集数据：去公司官网/SEC 下载 5 年的年报
3. 提取关键指标：营收、净利润、毛利率
4. 分析趋势：计算同比增长率，识别拐点
5. 生成报告：按"摘要→数据→分析→结论"结构写
```

先规划，再执行。**规划把模糊的大目标转化为清晰的子步骤序列**。

## 2. Plan-and-Execute 架构

### 2.1 基本思想

**Plan-and-Execute** 把 Agent 的工作分成两个明确的阶段：

1. **Plan 阶段**：LLM 根据用户任务，生成一个完整的执行计划（子步骤列表）
2. **Execute 阶段**：按计划逐步执行，每步可以用 ReAct 的方式（Thought-Action-Observation）

```
用户任务
  │
  ▼
┌──────────────┐
│  Planner     │  LLM 生成计划：
│  (规划器)     │  Step 1: ...
│              │  Step 2: ...
│              │  Step 3: ...
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  Executor    │  按计划逐步执行
│  (执行器)     │  每步用 ReAct
│              │  
└──────┬───────┘
       │
       ▼
  最终输出
```

### 2.2 一个完整示例

```
用户："帮我比较 React 和 Vue 框架，写一篇技术对比文章。"

Plan 阶段（Planner LLM 输出）：
  1. 搜索 React 的核心特性和优缺点
  2. 搜索 Vue 的核心特性和优缺点
  3. 搜索两者的性能对比数据
  4. 搜索两者的社区生态和使用趋势
  5. 整理对比维度：学习曲线、性能、生态、适用场景
  6. 撰写对比文章

Execute 阶段：
  [执行 Step 1]
  Thought: 搜索 React 的特性。
  Action: search("React framework features pros cons 2025")
  Observation: React 使用虚拟 DOM...

  [执行 Step 2]
  Thought: 搜索 Vue 的特性。
  Action: search("Vue framework features pros cons 2025")
  Observation: Vue 3 使用 Composition API...

  ...
  [所有步骤执行完毕后，生成最终文章]
```

### 2.3 Plan-and-Execute vs ReAct

| 对比 | ReAct | Plan-and-Execute |
|------|-------|-----------------|
| 规划时机 | 每一步临时决定 | 先生成完整计划 |
| 适合任务 | 短任务（2-5 步） | 长任务（5+ 步） |
| 灵活性 | 高（随时调整） | 低（需要修改计划的机制） |
| 目标保持 | 容易遗忘 | 计划本身就是"提醒" |
| 实现复杂度 | 简单 | 需要 Planner + Executor |

### 2.4 动态重规划

静态计划的问题：执行过程中可能发现原始计划不可行（比如某个 API 挂了、搜索不到需要的信息）。

**动态重规划**在每一步执行完后，Planner 可以根据结果**修改剩余计划**：

```
原始计划：
  1. 从 API A 获取数据 ✅ 已完成
  2. 从 API B 获取补充数据 ❌ API B 不可用
  3. 分析数据
  4. 生成报告

重规划后：
  2'. 从 API C 获取补充数据（替代 API B）
  3. 分析数据
  4. 生成报告
```

这是 Plan-and-Execute 的核心改进——**计划不是一成不变的，而是可以根据执行反馈动态调整的**。

## 3. 自我反思：Reflexion

### 3.1 问题：Agent 会犯错但不会从错误中学习

ReAct 和 Plan-and-Execute 有一个共同问题：**Agent 犯了错，如果不被人类纠正，下次还会犯同样的错**。

比如 Agent 在一个编程任务中用了错误的 API 调用方式，失败后它可能换一种方式重试——但这种"重试"没有系统性的反思过程，更像是随机尝试。

### 3.2 Reflexion 框架

**Reflexion**（Shinn et al., 2023）在 ReAct 的基础上加入了**显式的反思步骤**：

```
循环：
  1. 执行任务（用 ReAct 或 Plan-and-Execute）
  2. 评估结果（成功了吗？哪里出了问题？）
  3. 反思（这次失败的原因是什么？下次应该怎么改？）
  4. 把反思结论存入记忆
  5. 用更新后的记忆重试任务
```

一个编程任务的例子：

```
[第一次尝试]
任务：写一个函数，判断字符串是否是回文。
Agent 代码：
  def is_palindrome(s):
      return s == s[::-1]
测试结果：❌ "A man, a plan, a canal: Panama" 判断为 False

[反思]
失败原因：没有处理空格、标点和大小写。
教训：回文判断需要先清理字符串——去除非字母字符，统一转小写。

[第二次尝试（带着反思记忆）]
Agent 代码：
  def is_palindrome(s):
      cleaned = ''.join(c.lower() for c in s if c.isalnum())
      return cleaned == cleaned[::-1]
测试结果：✅ 通过
```

### 3.3 Reflexion 的关键设计

**反思不是简单的重试**。ReAct 遇到失败可能只是换个行动重试（类似于随机搜索）。Reflexion 要求 Agent 明确回答：

1. **哪里出了问题？**（诊断）
2. **根本原因是什么？**（分析）
3. **下次应该怎么做？**（改进策略）

这些反思结论被存入一个**经验记忆**（episodic memory），在下次尝试时作为上下文提供给 LLM。

```
System prompt 中加入：
"之前的尝试和反思：
 - 尝试 1 失败：原因是未处理特殊字符。教训：先清理输入。
 请基于以上经验改进你的方案。"
```

> 可靠程度：Level 2
>
> 参考：[Shinn et al., 2023 - Reflexion](https://arxiv.org/abs/2303.11366)

## 4. 任务分解策略

### 4.1 自顶向下分解

最直接的方式：让 LLM 把大任务拆成子任务。

```
Prompt: "请把以下任务分解为 3-7 个清晰的子步骤：
        [用户任务描述]"
```

这种方式简单有效，但 LLM 可能产生不合理的分解——比如子步骤之间有隐含依赖但没标出来，或者子步骤粒度不均匀。

### 4.2 分解质量的关键

好的任务分解满足：
- **完备**：子步骤覆盖了完成任务所需的所有工作
- **有序**：依赖关系清晰，知道哪些可以并行、哪些必须串行
- **可执行**：每个子步骤足够具体，Agent 知道怎么做
- **粒度适中**：不能太粗（"分析数据"太模糊）也不能太细（"打开浏览器"太琐碎）

### 4.3 失败后的策略

当某个子步骤失败时，Agent 有三个选择：

1. **重试**：用不同的方式重新执行当前步骤
2. **跳过**：这个步骤不是必需的，继续下一步
3. **重规划**：修改整个剩余计划

选择哪个策略取决于失败的性质——暂时性错误（API 超时）可以重试，信息缺失可以跳过或换源，根本性错误（任务定义有问题）需要重规划。

## 5. 规划的本质挑战

### 5.1 LLM 的规划能力有限

一个重要的事实：**LLM 的规划能力远不如它的语言理解能力**。

LLM 在预训练中学到的是"文本接龙"——预测下一个 token。生成一个合理的计划需要：预见未来的状态、评估不同路径的可行性、处理约束和依赖——这些能力 LLM 并不天然具备，更多是通过 prompt 工程"引导"出来的。

实验证据：在需要多步规划的基准（如 AlfWorld 家务任务、WebArena 网页操作）上，即使最强的 LLM，成功率也远低于专门的规划算法。

### 5.2 规划 vs 推理

| 对比 | 推理（CoT） | 规划 |
|------|-----------|------|
| 时间方向 | 回顾已有信息 | 预见未来步骤 |
| 不确定性 | 低（信息已给出） | 高（不知道执行结果） |
| 反馈 | 通常无反馈 | 执行结果提供反馈 |
| LLM 擅长度 | 较强 | 较弱 |

这就是为什么目前的 Agent 大多采用"**短规划 + 频繁反馈**"的策略——不做太远的规划，每执行一两步就根据结果调整，而不是一次性规划 20 步然后盲目执行。

> 可靠程度：Level 3（LLM 规划能力的局限性是活跃研究方向）

---

### 公式速查卡

本章无数学公式，核心是规划策略对比：

| 策略 | 含义 |
|------|------|
| Plan-and-Execute | 先用 Planner 生成完整计划，再用 Executor 逐步执行 |
| 动态重规划 | 执行过程中根据反馈修改剩余计划 |
| Reflexion | 执行→评估→反思→记住教训→重试，从错误中显式学习 |
| 自顶向下分解 | 让 LLM 直接把大任务拆成子步骤列表 |
| 短规划 + 频繁反馈 | 不做太远的规划，每步根据执行结果调整，避免盲目规划 |

    08 - 评估与调试：Agent 的测试方法论¶

    主维度：D3 工程实现 关键关系： - 评估 (任务) --用于--> Agent (架构)：评估用于衡量 Agent 的能力 - Benchmark (实验) --用于--> 评估 (任务)：Benchmark 用于标准化评估

    学习路径：... → 工程实现 → 本章（评估与调试） → 应用与前沿

    前置知识： - Agent 架构模式（参见 06-architectures.md） - Agent 工程实现（参见 07-engineering.md）

    参考： - Liu et al., 2023 - AgentBench - Zhou et al., 2023 - WebArena - Jimenez et al., 2024 - SWE-bench

    核心问题¶

    怎么知道你的 Agent 到底好不好用？

    Agent 比传统软件难测试得多。传统软件：输入确定 → 输出确定 → 对比即可。Agent：同一个输入可能走不同的路径、调用不同的工具、得到不同（但都合理）的结果。加上 LLM 的随机性，两次运行同一个任务的结果可能完全不同。

    可靠程度：Level 2-3（Agent 评估方法仍在快速发展中）

    1. Agent 评估的挑战¶

    1.1 为什么传统测试方法不够¶

    挑战 传统软件 Agent
    确定性 相同输入 → 相同输出 相同输入 → 可能不同路径和输出
    正确性判断 精确匹配期望值 多个回答都可能"正确"
    中间过程 不关心（黑盒测试） 过程也很重要（选对了工具吗？）
    外部依赖 可以 mock 工具调用结果可能随时间变化
    成本 跑一次测试几乎免费 每次测试都消耗 LLM API 费用

    1.2 评估什么¶

    Agent 的评估需要从三个层面考虑：

    结果质量：最终回答是否正确、完整、有用？

    过程质量：Agent 选择的路径是否合理？有没有多余的工具调用？有没有走弯路？

    效率：用了多少 token？调用了多少次 LLM？花了多长时间？

    一个 Agent 可能给出正确答案但过程极其低效（调用了 20 次搜索，其中 15 次是重复的），或者过程很漂亮但最终答案有误——两种情况都需要被识别和改进。

    2. 评估方法¶

    2.1 人工评估¶

    最直接的方法：人类看 Agent 的输出并打分。

    优点：判断最准确，能评估"有用性""自然度"等主观质量。 缺点：慢、贵、不可大规模重复。

    适合场景：开发早期，用 10-20 个测试用例人工检查 Agent 行为，快速发现明显问题。

    2.2 自动化评估（LLM-as-Judge）¶

    用另一个 LLM来评估 Agent 的输出质量。

    评估 Prompt:
"请评估以下 AI 助手的回答质量，打分 1-5。
 评估维度：准确性、完整性、相关性。

 用户问题：{question}
 助手回答：{agent_answer}
 参考答案：{reference_answer}（如有）

 请给出每个维度的评分和理由。"

    LLM-as-Judge 的可靠性取决于评估模型的能力——通常需要用比被评估 Agent 更强的模型来做评审。

    2.3 Benchmark 评估¶

    用标准化的任务集来测试 Agent，每个任务有明确的成功标准。

    3. 主流 Agent Benchmark¶

    3.1 代码类¶

    SWE-bench（Jimenez et al., 2024）：从 GitHub 上真实的 issue 和 PR 中构建的基准。给 Agent 一个 bug 报告，让它修复代码并通过测试。

    • 任务数量：2294 个真实的 GitHub issue
    • 评估标准：修改后的代码能否通过测试用例
    • 难度：截至 2025 年底，最好的 Agent 在完整版上的解决率约 50%

    HumanEval：代码生成基准，给函数签名和 docstring，生成函数体。

    3.2 网页操作类¶

    WebArena（Zhou et al., 2023）：在真实网站环境（Reddit、GitLab、购物网站等的镜像）中执行任务。

    • 任务示例："在论坛上发一个标题为 X 的帖子"、"在购物网站上找到最便宜的 Y 商品"
    • 评估标准：网站状态是否发生了预期的变化
    • 难度：最好的 Agent 成功率约 35-40%

    3.3 通用能力类¶

    AgentBench（Liu et al., 2023）：覆盖 8 种不同环境（操作系统、数据库、知识图谱、网页等）的综合基准。

    GAIA：Google 设计的通用 Agent 基准，覆盖各种实际任务。

    3.4 Benchmark 的局限¶

    • 不代表真实使用场景：Benchmark 的任务是精心设计的，真实用户的需求更模糊、更多样
    • 可能被过度优化：针对 Benchmark 调优的 Agent 在真实场景中可能表现不佳
    • 更新滞后：Agent 能力快速发展，Benchmark 可能很快被"做满"
    • 不测量用户体验：交互速度、回答风格、错误恢复等实际重要的维度不在测量范围内

    4. 调试方法¶

    4.1 Agent 常见的失败模式¶

    失败模式 表现 常见原因
    无限循环 Agent 反复调用同一个工具 工具返回的信息不够 Agent 做判断；缺少终止条件
    选错工具 该搜索时去算、该算时去搜 工具描述不清晰
    参数错误 工具调用参数格式不对 参数 schema 定义不够明确
    幻觉 不调用工具就瞎编答案 没有强制 Agent 必须用工具验证事实
    目标偏移 做着做着偏离了原始任务 长对话中上下文太多，原始目标被淹没
    过早终止 只完成了一部分就停止 任务分解不充分，Agent 以为做完了

    4.2 调试工作流¶

    1. 记录完整 trace
   ↓
2. 定位失败点：在哪一步出了问题？
   ↓
3. 分析原因：是 LLM 推理错误？工具问题？prompt 问题？
   ↓
4. 针对性修复
   ↓
5. 回归测试：修复后原来成功的用例是否仍然成功？

    4.3 Trace 记录¶

    Trace（执行轨迹）是 Agent 调试的核心工具。一个完整的 trace 记录每一步的：

    • 输入：LLM 收到的完整消息列表
    • 输出：LLM 的回复（文本或工具调用）
    • 工具执行：调用了什么工具、传了什么参数、返回了什么
    • 耗时：每一步花了多长时间
    • token 使用：每次 LLM 调用的 token 数
    [Step 1] LLM 调用 (1.2s, 450 tokens)
  输入: system + user message
  输出: tool_call(search, {"query": "..."})

[Step 2] 工具执行 (0.8s)
  search("...") → "结果..."

[Step 3] LLM 调用 (0.9s, 380 tokens)
  输入: 之前所有消息 + 工具结果
  输出: "最终回答..."

总计: 3 步, 2.9s, 830 tokens

    LangSmith（LangChain 的配套工具）提供了可视化的 trace 查看界面，在调试复杂 Agent 时非常有用。

    4.4 常见的修复手段¶

    问题 修复手段
    选错工具 改进工具 description，加入"什么时候该用/不该用"的说明
    参数错误 给参数加更详细的 description 和示例值
    无限循环 设置 max_turns；在 prompt 中加"如果连续 3 次搜索未找到，请直接回答"
    幻觉 在 prompt 中加"必须基于工具返回的信息回答，不要编造"
    目标偏移 在 system prompt 中用加粗/重复强调核心任务

    公式速查卡¶

    概念 含义
    LLM-as-Judge 用另一个 LLM 评估 Agent 输出质量
    SWE-bench 代码修复 Benchmark，基于 GitHub 真实 issue
    WebArena 网页操作 Benchmark，在真实网站镜像中执行任务
    Trace Agent 的完整执行轨迹（每步输入、输出、工具调用、耗时、token）
    回归测试 修复问题后验证原来成功的用例仍然成功

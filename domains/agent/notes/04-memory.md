## 核心问题

**LLM 的上下文窗口有限（比如 128K token），对话太长或知识太多怎么办？**

纯 LLM 的"记忆"就是上下文窗口——你发给它的所有文本。窗口满了，旧内容就丢了。Agent 需要一个更强大的记忆系统，来管理短期的任务上下文和长期的知识积累。

> 可靠程度：Level 1-2

---

## 1. 记忆的分类

### 1.1 短期记忆 vs 长期记忆

借用认知科学的术语，Agent 的记忆分两种：

| 类型 | 认知类比 | Agent 中的实现 | 特点 |
|------|---------|--------------|------|
| **短期记忆** | 工作记忆（你正在想的事） | LLM 的上下文窗口 | 容量有限，任务结束即丢失 |
| **长期记忆** | 长期记忆（你学过的知识） | 外部存储（向量数据库、文件） | 持久化，需要检索才能使用 |

### 1.2 短期记忆的管理

短期记忆就是 LLM 的上下文窗口。Agent 在执行多步任务时，对话历史会越来越长——每一轮 Thought-Action-Observation 都在累积 token。

当上下文接近窗口上限时，常见的管理策略：

**滑动窗口**：保留最近 $k$ 轮对话，丢弃最早的。简单但粗暴——早期的关键信息可能被丢掉。

**摘要压缩**：让 LLM 把较早的对话历史**压缩成摘要**，只保留关键信息。例如 10 轮对话压缩成一段 200 token 的摘要。

```
[原始对话 1-8 的摘要：用户想查找 2024 年 AI 论文，
 已搜索 arXiv 发现 3 篇相关论文，正在阅读第 2 篇...]

[完整对话 9]
[完整对话 10]
[当前用户输入]
```

**选择性保留**：根据相关性保留最重要的历史片段，而不是简单地按时间顺序截断。

### 1.3 长期记忆的必要性

短期记忆在以下场景不够用：

- **跨对话连续性**：用户今天和 Agent 的对话，明天继续时 Agent 应该记得之前的进展
- **大规模知识库**：把一本 500 页的书放进上下文窗口不现实（且成本极高）
- **个性化**：Agent 应该记住用户的偏好、常用工具、历史习惯

这些都需要**长期记忆**——把信息存储在 LLM 的上下文之外，需要时再检索回来。

## 2. RAG：检索增强生成

### 2.1 核心思想

**RAG**（Retrieval-Augmented Generation，检索增强生成，Lewis et al., 2020）是目前 Agent 长期记忆的核心技术。思路很简单：

> 不要把所有知识塞进上下文，而是在需要时**检索**最相关的部分放进来。

就像你写论文时不会把所有参考文献背下来，而是需要引用时去查——RAG 让 Agent 做同样的事。

### 2.2 RAG 的完整流程

RAG 分两个阶段：**索引阶段**（离线，只做一次）和**检索-生成阶段**（在线，每次查询时执行）。

**索引阶段**：

```
原始文档（PDF、网页、代码...）
  │
  ▼
1. 文本提取：把各种格式转成纯文本
  │
  ▼
2. 分块（Chunking）：把长文本切成小块（通常 200-1000 token/块）
  │
  ▼
3. 嵌入（Embedding）：用 embedding 模型把每个文本块转成一个高维向量
   例如 OpenAI text-embedding-3-small 输出 1536 维向量
  │
  ▼
4. 存入向量数据库：向量 + 原始文本一起存储
```

**检索-生成阶段**：

```
用户问题："Transformer 的注意力机制是怎么计算的？"
  │
  ▼
1. 嵌入查询：用同一个 embedding 模型把问题转成向量
  │
  ▼
2. 向量检索：在数据库中找到与问题向量最相似的 top-k 个文本块
   （用余弦相似度或点积衡量相似性）
  │
  ▼
3. 组装 Prompt：
   "请根据以下参考资料回答问题：
    [检索到的文本块 1]
    [检索到的文本块 2]
    ...
    问题：Transformer 的注意力机制是怎么计算的？"
  │
  ▼
4. LLM 生成回答（基于检索到的上下文）
```

### 2.3 为什么用向量检索而不是关键词搜索？

传统的关键词搜索（如 Ctrl+F）是精确匹配——查询 "注意力" 只能找到包含 "注意力" 这个词的文档。

**向量检索**（也叫语义检索）匹配的是**语义**：

- 查询 "注意力机制" 能找到包含 "attention mechanism" 的英文段落
- 查询 "Transformer 怎么处理长距离依赖" 能找到讲 self-attention 的段落（即使段落中没有 "长距离依赖" 这几个字）

这是因为 embedding 模型把**语义相近的文本映射到向量空间中相近的位置**——这正是词嵌入（`domains/LLM/notes/03-word-embedding.md`）的推广，只不过从词级别扩展到了段落/文档级别。

**余弦相似度**是衡量两个向量相似性的标准方法：

$$\text{sim}(\mathbf{q}, \mathbf{d}) = \frac{\mathbf{q} \cdot \mathbf{d}}{|\mathbf{q}||\mathbf{d}|} = \frac{\sum_{i=1}^{n} q_i d_i}{\sqrt{\sum_{i=1}^{n} q_i^2} \cdot \sqrt{\sum_{i=1}^{n} d_i^2}}$$

其中 $\mathbf{q}$ 是查询向量，$\mathbf{d}$ 是文档向量，$n$ 是向量维度。值域为 $[-1, 1]$，越接近 1 表示越相似。

**数字例子**：假设 $\mathbf{q} = (1, 0, 1)$，$\mathbf{d} = (1, 1, 0)$：

$$\text{sim} = \frac{1 \times 1 + 0 \times 1 + 1 \times 0}{\sqrt{1^2+0^2+1^2} \cdot \sqrt{1^2+1^2+0^2}} = \frac{1}{\sqrt{2} \cdot \sqrt{2}} = \frac{1}{2} = 0.5$$

0.5 表示"有一定相似性但不完全匹配"。如果 $\mathbf{d} = (1, 0, 1)$（和查询完全相同），sim = 1。

> 参考：[Lewis et al., 2020 - RAG](https://arxiv.org/abs/2005.11401)

## 3. 向量数据库

### 3.1 是什么

**向量数据库**是专门用于存储和检索高维向量的数据库。和传统数据库（MySQL、PostgreSQL）的区别：

| 特性 | 传统数据库 | 向量数据库 |
|------|-----------|-----------|
| 存储 | 结构化数据（表格、行列） | 高维向量 + 原始数据 |
| 查询方式 | SQL（精确匹配） | 近似最近邻搜索（ANN） |
| 典型查询 | `WHERE name = '张三'` | "找到与此向量最相似的 10 个" |
| 用途 | 业务数据管理 | 语义搜索、推荐、RAG |

### 3.2 主流向量数据库

| 名称 | 特点 | 适用场景 |
|------|------|---------|
| **Chroma** | 轻量级，Python 原生，可嵌入应用 | 原型开发、小规模项目 |
| **Pinecone** | 云托管，全管理，开箱即用 | 生产环境、不想自己运维 |
| **Weaviate** | 开源，内置 embedding 生成 | 需要灵活定制 |
| **FAISS** | Meta 开源的向量搜索库（不是数据库） | 需要极高性能的大规模检索 |
| **Milvus** | 开源，分布式，支持十亿级向量 | 大规模生产环境 |

对于学习和原型开发，**Chroma** 是最简单的入门选择。

### 3.3 近似最近邻搜索（ANN）

在百万级甚至十亿级的向量中，逐一计算余弦相似度太慢。向量数据库使用**近似最近邻搜索**（Approximate Nearest Neighbor, ANN）算法：

- 不保证找到绝对最相似的向量，但能在毫秒级返回"足够相似"的结果
- 常见算法：HNSW（分层导航小世界图）、IVF（倒排文件索引）
- 在 RAG 场景中，"近似"已经足够好——top-10 中偶尔少一两个最相关的，对最终回答影响不大

## 4. 分块策略

### 4.1 为什么需要分块

Embedding 模型有输入长度限制（通常 512-8192 token），而文档通常远超这个长度。即使模型能处理长文本，**越短的文本块，embedding 越精确**——一整本书的 embedding 太笼统，无法区分书中不同章节的内容。

### 4.2 常见分块方法

**固定大小分块**：每 500 token 切一块，相邻块之间有 50 token 重叠（防止关键信息被切断）。简单但可能切断语义完整的段落。

**按语义结构分块**：按标题、段落、章节等自然边界切分。保持语义完整，但需要文档有清晰的结构。

**递归分块**：先按大结构（章节）切分，如果某块太长，再按小结构（段落、句子）递归细分。LangChain 的 `RecursiveCharacterTextSplitter` 就是这种策略。

### 4.3 分块大小的权衡

| 小块（~200 token） | 大块（~1000 token） |
|-------------------|-------------------|
| 检索精度高（匹配更精确） | 上下文更完整 |
| 可能缺少上下文（一句话脱离前后文不好理解） | 检索精度低（不相关内容被一起检索到） |
| 需要检索更多块来凑够信息 | 占用更多上下文窗口空间 |

实践中通常 **300-800 token**，加 **10-20% 重叠**，是比较好的起点。

## 5. RAG 的局限与改进

### 5.1 RAG 的常见失败模式

| 失败模式 | 原因 | 表现 |
|---------|------|------|
| **检索不到** | 查询和文档的表述方式差异太大 | Agent 回答"我找不到相关信息" |
| **检索到但不相关** | 向量相似但语义不匹配 | Agent 给出偏题的回答 |
| **检索到但回答错** | LLM 没有正确理解检索到的上下文 | 有正确信息但回答仍然错 |
| **信息分散** | 答案需要综合多个文档块 | Agent 只用了其中一个块 |

### 5.2 改进方向

**查询改写**（Query Rewriting）：先让 LLM 把用户的原始查询改写成更适合检索的形式，再去搜索。

```
用户原始查询："这个 bug 怎么修？"
改写后查询："Python TypeError: unhashable type 解决方案"
```

**混合检索**：同时使用向量检索（语义）和关键词检索（精确），合并结果。处理专有名词或代码时关键词检索更有效。

**重排序**（Re-ranking）：先用向量检索得到 top-50 候选，再用一个更精确的模型对候选重新排序，取 top-5。

**多步检索**：第一次检索后，根据结果生成新的查询，再检索一次。类似 ReAct 的多步搜索模式。

> 可靠程度：Level 2-3

| 公式/概念 | 含义 |
|----------|------|
| $\text{sim}(\mathbf{q}, \mathbf{d}) = \frac{\mathbf{q} \cdot \mathbf{d}}{\|\mathbf{q}\|\|\mathbf{d}\|}$ | 余弦相似度，衡量两个向量的语义相似性，值域 $[-1, 1]$ |
| Embedding | 把文本映射到高维向量空间，语义相近的文本距离近 |
